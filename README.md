# Git Flow Guard

Temporary development repository for a Mermaid `gitGraph` to Git `reference-transaction` hook policy workflow.

Current focus:

- Treat a restricted `CONTRIBUTING.md` Mermaid `gitGraph` as the human-authored policy source.
- Parse branch-family edges such as `"infra/*" -> dev`.
- Generate policy that can drive a local `reference-transaction` hook.
- Provide a Codex skill for writing compatible `CONTRIBUTING.md` gitGraph documents.

## Install Package

Install the package from this repository:

```bash
python -m pip install -e .
```

This exposes one console command:

```text
git-flow-guard
```

## Configs

Each config lives under `configs/<name>/` and contains:

- `contribution.md`: the human-authored Mermaid `gitGraph` source of truth.
- `policy.yaml`: generated policy output. Do not edit this file by hand.
- `test_case.py`: config-specific hook behavior tests for this policy.

Shared test scaffolding lives in `configs/test_base.py`. It contains only generic repo setup, hook installation, Git command helpers, and rejection assertions. Policy-specific behavior such as creating a valid DAG or choosing rejected operations belongs in each config's `test_case.py`.

Generate all bundled policies:

```bash
git-flow-guard generate --all
```

Check that generated policies are current:

```bash
git-flow-guard generate --all --check
```

Generate one config:

```bash
git-flow-guard generate configs/infra-feat-release
```

The parser intentionally supports a small DSL:

- `branch NAME`: records a branch-from edge from the current checkout.
- `checkout NAME`: changes the current target context.
- `merge NAME id:"SOURCE to TARGET" tag:"..."`: records a merge rule into the current checkout.

`tag:"..."` is optional. In tag patterns, `#` means one or more decimal digits, and `=` means the same numeric component as the source branch's base release tag.

The first bundled skill lives at:

```text
.codex/skills/git-flow-policy-writer/SKILL.md
```

## Install Hook

Install one config into a target repository:

```bash
git-flow-guard install --repo /path/to/repo --config infra-feat-release --scope worktree
```

`--config` accepts either a bundled config name under `configs/`, a config directory containing `contribution.md`, or a direct `contribution.md` path.

`--scope` controls where `git config core.hooksPath .git-flow-guard/hooks` is written:

- `worktree`: writes to this worktree's config. This is the default and is best for multi-worktree development.
- `local`: writes to the repository-local config.
- `global`: writes to the user's global Git config.

The installer copies the packaged runtime hook runner into the target repository under `.git-flow-guard/runtime/` and writes the actual `reference-transaction` hook under `.git-flow-guard/hooks/`. Because the configured hook path is repo-relative, a test repo created in Docker can still run the same hook from the host without referencing container-only paths such as `/workspace/...`.

## Test Env

`test_env/` contains the Docker image plus one batch runner, `test_env/run_policy_hook_tests.py`. The runner discovers `configs/*/test_case.py` and executes each `TEST_CASE`.

Build:

```bash
mkdir -p .tmp
docker compose build policy-hook-tests
```

Run:

```bash
mkdir -p .tmp
docker compose run --rm policy-hook-tests
```

The policy hook test runner creates one isolated Git repository per config under `.tmp/<config_name>`.
